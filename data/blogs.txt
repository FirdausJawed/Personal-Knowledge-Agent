1. Garbage Collection (GC)
Don't start with algorithms. Start with intent.
What interviewers really want to hear:
Why GC exists and what problem it solves in real systems.
GC exists so developers don't manually manage memory and accidentally crash long-running services with leaks or dangling references. In backend systems, that safety matters more than raw performance.
Once you've said that, then talk about how it works:
Objects get allocated on the heap
Reachability decides whether an object is alive
Collectors reclaim unused memory

Now pause and ask yourself:
👉 Can you explain why GC tuning matters in high-traffic services?
Follow-up they might ask:
Why would a poorly tuned GC cause latency spikes even if CPU usage looks fine?
If you can answer that, you're thinking like an SDE-2 already.

---

2. HashMap Internals (This One Decides Interviews)
HashMap questions are never random.
Interviewers ask them because HashMap exposes how you reason about performance under collision.
You should be able to walk through:
How key → hash → bucket mapping works
What happens during collisions
Why Java introduced treeification

But here's the real signal:
Treeification doesn't happen immediately. It happens only after a collision threshold and minimum capacity.
That detail separates memorization from understanding.
Now think about this:
👉 In a high-traffic API, what kind of bug could cause excessive HashMap collisions even with a large capacity?
If you say poor hashCode implementation, you're on the right track.

---

3. Streams API (Lazy Execution Is the Hook)
Most candidates say: Streams make code cleaner. That's weak.
Streams exist to:
Separate what you want from how it executes
Enable lazy evaluation
Allow internal optimizations

You should clearly explain Why intermediate operations don't execute immediately, When execution actually starts, How map vs flatMap changes the data shape
Try answering this:
👉 When you chain filter → map → collect, at what exact point does iteration begin?
If you hesitate here, revise streams again.

4. Functional Interfaces (Small Topic, Big Signal)
This is a trap question.
Interviewers don't care if you can define a functional interface. They care if you understand why Java needed it.
Functional interfaces:
Enable lambda expressions
Allow behavior to be passed, not just data
Make APIs like Streams possible

Follow-up they might throw:
👉 Why can't a functional interface have two abstract methods?
Answering that cleanly shows conceptual clarity.

---

5. Object Copying (Shallow vs Deep)
This one shows whether you've debugged real bugs.
You must clearly explain:
Shallow copy copies references
Deep copy duplicates the object graph

But don't stop there.
Connect it to reality:
👉 Why can shallow copies cause bugs in multi-threaded systems?
If you mention shared mutable state, you're thinking correctly.

---

6. Strings (Immutability Is Not Just "Safety")
Everyone says:
Strings are immutable for security.
That's incomplete.
Immutability also Enables string pooling, Makes Strings thread-safe by design, Allows JVM optimizations
You should confidently explain:
What the String pool is
How interning works
Why == behaves differently from equals()

Now answer this honestly:
👉 In what real scenario would using == accidentally pass tests but fail in production?
That's an interview-level follow-up.

---

7. OOP - Keep It Structured, Not Long
Interviewers don't want essays here.
State the four pillars:
Encapsulation
Abstraction
Inheritance
Polymorphism

Then give one real backend example, not definitions.
Example:
Encapsulation prevents external services from mutating internal state directly.
If you over-explain OOP, you sound junior. Structure beats verbosity.
